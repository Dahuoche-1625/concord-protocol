# Concord Protocol v0.2 Roadmap

> Status: planning-draft
> Goal: Validate and harden the Bridge Object loop before expanding runtime power.

---

## 1. Scope Principle

v0.2 is not a feature expansion release.

v0.2 exists to harden what v0.1 already introduced:

```text
TaskContract -> TaskDispatch -> TaskLease -> ExecutionReceipt -> ReviewResult -> AuditLog
```

The release should resolve known v0.1 debts, validate the bridge flow with one real project task, and avoid expanding Concord into a full orchestration framework.

---

## 2. v0.2 Target

```text
Validate and harden the Bridge Object loop before expanding runtime power.
```

v0.2 should prove that:

- bridge objects cannot be silently forged
- task contracts are clear enough to execute
- worker binding happens through lease claim
- receipts prove artifacts
- review decisions are auditable
- version-incompatible agents are not dispatched
- guard behavior is stronger than `verify_only` without pretending to be an OS sandbox

---

## 3. Guard Mode Decision

### 3.1 Do Not Claim Full `enforced` Yet

`guard_mode: enforced` requires a real enforcement stack:

```text
filesystem interception
sandbox / container / wrapper
permission isolation
runtime-level deny before write
```

Adding the string `"enforced"` without those mechanisms creates false security.

### 3.2 v0.2 Mode: `guarded_verify`

v0.2 should introduce:

```text
guard_mode: guarded_verify
```

Meaning:

- pre-execution validation of identity, boundary, task mode, lease, and version
- post-execution workspace delta detection
- out-of-scope result invalidation
- automatic cleanup of newly created out-of-scope files when safe
- rollback of modified out-of-scope files only when a pre-task snapshot exists
- audit event for every cleanup or rollback attempt

`guarded_verify` is stronger than `verify_only`, but it is not a sandbox.

Full `enforced` mode is deferred until the protocol has a real OS-level or container-level reference implementation.

---

## 4. Must Do

These items must be completed before v0.2 can be considered valid.

| Item | Why it matters | Acceptance standard |
| --- | --- | --- |
| Bridge Object integrity checks | Prevent forged or tampered bridge objects from breaking domain separation | `source_proof`, `claim_proof`, and `execution_hash` fields defined; at least one forged object is rejected by guard/validator |
| Task model split | Separate local execution snapshots from distributed bridge contracts | `TaskSnapshot` and bridge-layer `TaskContract` are separate schemas; old generic `Task` shape is marked deprecated |
| TaskContract quality gate | Prevent vague contracts from causing execution bias | `acceptance_criteria`, `validation_script_ref`, and `contract_clarity_score` fields defined; at least one ambiguous contract is refused before execution |
| Protocol version compatibility | Prevent mixed-version agents from silently misreading tasks | `protocol_version_min/max` and task `required_protocol_version` defined; incompatible agent is rejected before dispatch or claim |
| Bridge Loop validation | Prove the protocol works on a real task | One pilot agent completes the full loop: TaskContract, Dispatch, Lease, Receipt, Review, AuditLog, artifact validation, and guard judgment |

---

## 5. Should Do

These items should be included if they directly support the must-do validation. They should not expand v0.2 beyond the bridge hardening goal.

| Item | Reason | Constraint |
| --- | --- | --- |
| `file_bus_v1` bridge adapter reference | Provides a minimal local bridge implementation | Must include atomic lease claim or be marked `single_worker_only` |
| Hermes pilot checklist | Gives one strict agent a repeatable validation script | Should run with one agent first, not four agents at once |
| Compliance measurement draft | Makes protocol adherence visible | Draft only; do not turn it into a scoring/governance subsystem yet |
| Cleanup/rollback policy | Makes `guarded_verify` behavior explicit | Must distinguish safe cleanup from unsafe rollback |

---

## 6. Later

These ideas are valid but should not be part of v0.2 unless the must-do list is already complete.

| Item | Why later |
| --- | --- |
| Full `enforced` sandbox | Requires OS/container-level enforcement, not just protocol fields |
| `feishu_bus_v1` / `http_bus_v1` / `queue_bus_v1` full adapters | Useful after local bridge loop is proven |
| Committee `quorum_rules` | Important governance work, but not required to validate bridge execution |
| `vote_weight` and tie-breaking | Risk of governance complexity before runtime basics are verified |
| `cooldown_required` | Good risk-control concept, but belongs after audit/compliance data exists |
| Full `compliance_score` system | Needs enough task data to define meaningful metrics |
| Cross-project memory sharing | Explicitly outside v0.2; risks weakening domain separation |

## 7. Proposed v0.2 Object Changes

### 7.1 Bridge Object Integrity

Add proof fields:

```json
{
  "source_proof": {
    "method": "hmac | signature | hash_chain",
    "key_ref": "project-key-ref",
    "signed_fields": ["task_id", "project_id", "required_capabilities", "input_refs"],
    "value": "..."
  }
}
```

```json
{
  "claim_proof": {
    "method": "hmac | signature | token",
    "node_id": "node-win-01",
    "agent_id": "worker-agent-id",
    "timestamp": "ISO8601",
    "value": "..."
  }
}
```

```json
{
  "execution_hash": {
    "method": "sha256",
    "artifact_refs_hash": "...",
    "receipt_hash": "..."
  }
}
```

Minimum implementation may use HMAC. Full PKI is not required for v0.2.

### 7.2 Task Model Split

Local mode:

```json
{
  "task_mode": "local",
  "task_snapshot_id": "TASK-SNAPSHOT-0001",
  "task_id": "TASK-0001",
  "assigned_agent": "agent-alpha",
  "lease_id": "LEASE-0001"
}
```

Bridge mode:

```json
{
  "task_mode": "bridge",
  "task_id": "TASK-0001",
  "required_capabilities": ["video_pipeline_execution"],
  "target_mesh": "mesh-any",
  "assigned_agent": null
}
```

Rules:

- `TaskSnapshot` may contain `assigned_agent`.
- Bridge-layer `TaskContract` should not bind a concrete worker before `TaskLease`.
- RuntimeGuard must reject a distributed task that bypasses lease binding.

### 7.3 TaskContract Quality Gate

Add:

```json
{
  "acceptance_criteria": [
    "final artifact exists",
    "duration is within expected range",
    "receipt includes artifact_refs",
    "RuntimeGuard result is pass"
  ],
  "validation_script_ref": "validators/video_pipeline_receipt.py",
  "contract_clarity_score": 4
}
```

Rules:

- An agent may refuse execution when `contract_clarity_score < 3`.
- Refusing an ambiguous contract is not a protocol violation.
- ReviewResult should record whether acceptance criteria were fully checked.

### 7.4 Version Compatibility

FrameworkManifest:

```json
{
  "protocol_version_min": "0.1",
  "protocol_version_max": "0.2"
}
```

Agent manifest or identity registry:

```json
{
  "supported_protocol_versions": {
    "min": "0.1",
    "max": "0.2"
  }
}
```

TaskContract:

```json
{
  "required_protocol_version": "0.2"
}
```

Rules:

- Incompatible agents must not be dispatched.
- If incompatibility is discovered at claim time, claim must be rejected and audited.
- Unknown fields should be tolerated only when the task's required version is within the agent's supported range.

## 8. Pilot Validation Plan

Recommended pilot:

```text
One agent.
One low-risk task.
UPLOAD=false.
Real artifacts.
Full bridge loop.
```

Suggested pilot role:

```text
Hermes as strict worker/reviewer pilot
```

Pilot checklist:

1. Receive TaskContract and verify clarity.
2. Reject contract if clarity score is below threshold.
3. Claim TaskLease atomically.
4. Execute only with disclosed `input_refs`.
5. Produce artifact(s).
6. Write ExecutionReceipt with artifact refs and execution hash.
7. Run RuntimeGuard in `guarded_verify`.
8. Write ReviewResult.
9. Append audit events for every transition.

Success metrics:

- full bridge loop completed
- domain separation violations = 0
- forged bridge object rejected at least once in a test case
- ambiguous TaskContract rejected at least once in a test case
- incompatible protocol version rejected at least once in a test case
- artifact refs verified against real output

---

## 9. Non-Goals for v0.2

v0.2 does not need to solve:

- multi-transport production deployment
- full committee governance
- global reputation or compliance scoring
- OS-level sandboxing
- cross-project memory
- automatic business approval
- high-frequency distributed scheduling

These are later concerns. v0.2 should make the bridge loop trustworthy first.
