# Concord Protocol v0.2 Pilot Plan

> Status: pilot-draft
> Goal: Run one real Bridge Object loop before implementing broader v0.2 changes.
> Related: [v0.2 Roadmap](v0.2-roadmap.md), [Domain Separation Model](domain-separation-model-v0.1.md)

---

## 1. Pilot Goal

Validate this loop with one low-risk real task:

```text
TaskContract
-> TaskDispatch
-> TaskLease
-> ExecutionReceipt
-> ReviewResult
-> AuditLog
```

The pilot is not a scheduler, not a sandbox, and not a production deployment. It is a protocol validation run.

Success means the objects, transitions, integrity fields, artifact references, and review trail are coherent enough to support a v0.2 implementation.

---

## 2. Pilot Task

Recommended task:

```text
Run a local video pipeline dry-run for one existing drama/source folder.
UPLOAD=false.
Produce final artifact, pipeline log, execution receipt, review result, and audit log.
```

Constraints:

- no YouTube upload
- no production approval mutation
- no OAuth token movement
- no full project memory disclosure
- no direct Runtime access to project facts outside TaskContract.input_refs
- no direct Project invocation of a worker

## 4. Roles

### 4.1 Project Issuer

Creates or approves `TaskContract`.

Responsibilities:

- disclose only minimum context
- define acceptance criteria
- define output contract reference
- include source proof
- avoid concrete worker binding in bridge mode

### 4.2 Coordinator

Publishes `TaskDispatch`.

Responsibilities:

- verify project authority
- choose target mesh or dispatch policy
- not mutate project facts
- not directly invoke worker

### 4.3 Worker

Claims `TaskLease` and executes locally.

Responsibilities:

- claim lease atomically
- execute only with disclosed context
- produce artifacts and logs
- write ExecutionReceipt
- stop if lease renewal is not confirmed before expiry

### 4.4 Reviewer

Writes `ReviewResult`.

Responsibilities:

- check acceptance criteria
- verify artifact refs
- verify guard judgment
- decide accepted, rejected, retry_requested, or escalated

## 5. Step-by-Step Runbook

### Step 1 вЂ” Create TaskContract

Required checks:

- `task_mode = "bridge"`
- `assigned_agent = null`
- `required_capabilities` present
- `input_refs` minimal
- `redactions` present
- `acceptance_criteria` present
- `contract_clarity_score >= 3`
- `required_protocol_version` compatible
- `source_proof` present

If the contract is ambiguous, worker may refuse before execution.

### Step 2 вЂ” Publish TaskDispatch

Required checks:

- references the TaskContract
- has `target_mesh`
- has `dispatch_policy`
- status starts as `open`
- audit event appended

### Step 3 вЂ” Claim TaskLease

Required checks:

- claim is atomic for the transport
- no active lease exists for same task/output contract
- `lease_until` exists
- `claim_proof` exists
- audit event appended

If atomic claim is not available, adapter must be marked `single_worker_only`.

### Step 4 вЂ” Execute Locally

Required checks:

- worker uses only TaskContract input refs and deployment-local config
- no production upload
- no secret movement
- logs written under declared artifact/log scope
- long-running work renews lease before expiry

### Step 5 вЂ” Write ExecutionReceipt

Required checks:

- `receipt_id`, `task_id`, `lease_id`, `agent_id`, `node_id`
- `artifact_refs` present
- `log_refs` present
- `execution_hash` present
- status is `done`, `partial_done`, `failed`, or `invalid`
- audit event appended

### Step 6 вЂ” Run Guarded Verify

Required checks:

- identity valid
- boundary valid
- lease still valid or receipt is marked late/invalid
- protocol version compatible
- output refs in scope
- workspace delta checked
- newly created out-of-scope files cleaned up when safe
- modified out-of-scope files rolled back only when snapshot exists
- cleanup/rollback events audited

### Step 7 вЂ” Review Result

Required checks:

- all acceptance criteria checked
- artifact refs verified
- guard judgment included
- decision is explicit
- review references receipt
- audit event appended

## 6. Negative Tests

The pilot is incomplete unless these failures are tested.

| Test | Expected result |
| --- | --- |
| Tampered TaskContract source proof | reject before dispatch or claim |
| Fake TaskLease claim proof | reject claim |
| Receipt artifact hash mismatch | reject receipt or mark invalid |
| Ambiguous TaskContract clarity score `< 3` | worker refuses execution |
| Version mismatch | reject dispatch or claim |
| Out-of-scope artifact path | guarded_verify invalidates result |
| Expired lease writes completion receipt | reject receipt |

---

## 7. Acceptance Criteria

The pilot is successful only when:

- full bridge loop completes once with valid objects
- every state transition has an audit event
- artifact refs point to real outputs
- execution hash is reproducible
- ReviewResult references ExecutionReceipt
- at least one forged object is rejected
- at least one ambiguous contract is refused
- at least one version mismatch is rejected
- no project facts leak into runtime memory outside declared input refs

---

## 8. What Not To Do

- Do not start with four agents.
- Do not start with upload.
- Do not start with Feishu write-back.
- Do not implement full HTTP or queue transport first.
- Do not claim `guard_mode: enforced`.
- Do not migrate existing project directories before one small bridge loop passes.
